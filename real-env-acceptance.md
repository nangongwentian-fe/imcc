# imcc 真实环境验收手册

本文用于验证以下改动是否在真实飞书环境中按预期工作：

- Phase 0：Claude `stream-json` 流式链路
- Phase 1：飞书流式卡片输出
- Phase 2：飞书权限审批卡片
- Phase 3：本地会话列表与恢复

建议按本文顺序执行，不要跳测。

---

## 1. 验收前准备

### 1.1 本地启动

在项目根目录启动：

```bash
npm run dev
```

看到以下信息即可继续：

- `imcc starting (...)`
- `Lark: WebSocket long connection started`

### 1.2 飞书后台前置检查

至少确认以下配置已经完成：

1. 事件订阅已启用 `im.message.receive_v1`
2. 订阅方式为“长连接”
3. 应用版本已发布
4. 机器人能在目标私聊/群聊里收发消息
5. 已开通 `cardkit:card:write`

如果你本地配置了 `allowedUserIds`，还要确认当前发消息的飞书账号在白名单内。

### 1.3 建议同时观察终端

验收时保持终端窗口可见，重点关注这些异常日志：

- `[router]`
- `[lark] failed to create streaming card`
- `[lark] failed to finalize streaming card`
- `[lark] failed to send permission card`
- `[permission]`

## 2. Phase 0 + Phase 1：流式输出验收

### 用例 1：基础流式文本

在飞书发：

```text
请分 8 行输出 imcc 这个项目的作用，每行一句。
```

预期结果：

1. 消息发出后很快出现一张卡片，不应长时间无反馈
2. 初始内容应先出现 `Thinking...`
3. 卡片正文应持续更新，而不是最后一次性整段出现
4. 完成后卡片底部应显示状态与耗时
5. 终端不应出现未处理异常

失败判定：

- 一直只有 Typing，没有卡片
- 卡片直到结束才一次性出现完整内容
- 结束后卡片没有最终状态

### 用例 2：代码块渲染

在飞书发：

```text
输出一个 TypeScript 示例，包含一个 interface 和一个函数，用代码块包起来。
```

预期结果：

1. 卡片内代码块能正常渲染
2. 不应出现明显的 markdown 断裂
3. 完成态卡片正文应和最终回答一致

### 用例 3：错误链路

在飞书发一个明显会失败的请求，例如：

```text
请读取一个根本不存在的文件：/tmp/imcc-this-file-should-not-exist.txt，并告诉我结果。
```

预期结果：

1. 仍然先出现流式卡片
2. 最终卡片应以错误形式结束，而不是卡死
3. Typing 应被清掉

## 3. Phase 1：卡片降级验收（可选）

这一项无法在当前实现里直接开开关强制触发，只能用“没有 `cardkit:card:write` 的应用”或权限异常环境验证。

如果你能准备一个不带 `cardkit:card:write` 的飞书应用，按和上面相同的消息再测一次。

预期结果：

1. 流式卡片创建失败后，不应整条链路报废
2. 最终应退化为普通 `post` 消息回复
3. 终端应出现类似：

```text
[lark] failed to create streaming card, falling back to post:
```

如果你暂时没有第二个应用，这一项可以先记为“未验证”。

## 4. Phase 2：权限审批验收

## 4.1 单次 Allow

在飞书发：

```text
读取 src/bridge.ts，告诉我 runClaudeStream 做了哪 3 件事。
```

预期结果：

1. Claude 尝试调用工具时，应收到权限审批卡片
2. 卡片里应看到工具名和输入摘要
3. 点击 `Allow` 后，本次请求继续执行
4. 最终正常返回答案

失败判定：

- 没有审批卡片，也没有继续执行
- 点击后无响应
- 点击后 Claude 仍直接报权限拒绝

## 4.2 单次 Deny

再次发一个需要工具的请求：

```text
读取 src/router.ts，并告诉我消息流式结束时做了什么。
```

这次点击 `Deny`。

预期结果：

1. Claude 不应继续执行原工具调用
2. 最终应返回一个“权限被拒绝”或等价错误
3. 不应卡住超过 10 秒以上无结果

## 4.3 Allow Session

先发：

```text
读取 src/commands.ts，告诉我新增了哪些 slash 命令。
```

在审批卡片中点击 `Allow Session`。

然后立刻再发：

```text
再读取 src/sessions.ts，告诉我会话列表是怎么扫描出来的。
```

预期结果：

1. 第一个请求出现审批卡片
2. 第二个请求如果使用的是同一种工具，应该不再重复弹同类审批
3. 第二个请求能直接执行

说明：

- 当前实现是“按工具名加入当前 jid 的允许列表”
- 所以 `Allow Session` 的效果是“本会话内后续同工具自动放行”

## 4.4 审批超时

再发一个需要工具的请求：

```text
读取 README.md，并用 3 句话总结。
```

这次什么都不要点，等待 30 秒以上。

预期结果：

1. 审批请求应自动超时
2. Claude 最终应收到 deny 结果，而不是无限等待
3. 最终消息应明确失败，而不是静默消失

## 4.5 文本降级 `/perm`

这一项只在“卡片按钮不可用”时需要验。

如果你遇到以下情况：

- 按钮点了无效
- 根本没有审批卡片
- 终端出现 `failed to send permission card`

则按文本方式处理：

```text
/perm allow <permId>
```

或

```text
/perm allow-session <permId>
```

或

```text
/perm deny <permId>
```

如果你拿不到 `permId`，这项先记为“未验证”。当前实现的文本降级主要是保底链路，不是主验收路径。

## 5. Phase 3：会话列表与恢复验收

这一项建议严格按下面步骤做，否则很难判断到底是在用旧上下文还是新上下文。

### 用例 1：会话列表

先发：

```text
/sessions 5
```

预期结果：

1. 返回最近会话列表
2. 每条包含：
   - 会话 id 前缀
   - 相对时间
   - 项目路径
   - 首条消息摘要

### 用例 2：恢复指定会话

#### 第一步：制造一个可识别的旧上下文

先发：

```text
/clear
```

再发：

```text
记住口令 TEST-IMCC-RESUME-0318。只回复“已记住”。
```

确认机器人回复了“已记住”或等价表达后，再发：

```text
/sessions 5
```

记下刚才这个会话的 id 前缀，例如 `abc12345`。

#### 第二步：切到新会话验证“记忆丢失”

发：

```text
/clear
```

然后发：

```text
刚才让我记住的口令是什么？
```

预期结果：

1. 在新会话里，Claude 不应该稳定答出 `TEST-IMCC-RESUME-0318`
2. 如果它还能答出来，这一轮就不具备验证意义，换一个更随机的口令重测

#### 第三步：恢复旧会话

发：

```text
/resume <你刚才记下的id前缀>
```

然后再发：

```text
刚才让我记住的口令是什么？
```

预期结果：

1. `/resume` 应提示已切换到某个会话
2. 之后这条消息应在旧 session 上继续
3. Claude 应能答出：

```text
TEST-IMCC-RESUME-0318
```

失败判定：

- `/resume` 找不到会话
- `/resume` 后仍像新会话一样不知道口令

## 7. 常见失败点

### 7.1 发消息没反应

优先检查：

1. 当前账号是否在 `allowedUserIds`
2. 飞书应用是否已发布
3. `im.message.receive_v1` 是否已订阅
4. 长连接是否真的连上

### 7.2 有消息但没有流式卡片

优先检查：

1. 是否开通 `cardkit:card:write`
2. 终端是否出现：

```text
[lark] failed to create streaming card
```

3. 如果有，说明当前走了降级而不是流式卡片

### 7.3 审批卡片按钮点了没反应

优先检查：

1. 终端是否有报错
2. 飞书后台里和卡片交互/卡片回调相关的能力是否已开启
3. 长连接是否真的收到了 `card.action.trigger`

### 7.4 `/resume` 找得到会话但上下文没恢复

优先检查：

1. 你记录的 id 前缀是否对应正确会话
2. `/resume` 后是否又执行了 `/clear`
3. 恢复前后的工作目录是否变了
